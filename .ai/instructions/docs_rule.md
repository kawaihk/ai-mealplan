# ドキュメントレビュールール

## 概要
プロジェクトにおける要件定義資料や設計資料の品質を担保し、設計意図や改善プロセスを可視化するため、AIによるセルフレビューの実施と結果の記録をルール化します。

## ルール内容
1. **セルフレビューの実施**:
   - `docs` 配下の**全ての階層（サブディレクトリを含む）**にある資料（要件定義、設計書等）を新規作成または更新した際、必ずセルフレビューを実施すること。
   - **製造工程移行時の評価**: 製造工程（実装）へ移行する直前のレビューにおいては、実装者が迷うことなく作業に着手できるか、必要な資料（API仕様、非機能要件、ユースケース、画面遷移、ログ・メッセージ定義等）に過不足がないかを重要な評価指標として確認すること。
2. **レビュー結果の保存**:
   - レビュー結果は一貫して `docs/reviews/` ディレクトリに集約して保存する。
3. **ファイル形式と命名規則**:
   - ファイル形式: マークダウン形式 (`.md`)
   - ファイル名: `REV-YYYYMMDD-NN.md` の形式でレビューIDを付与する。
4. **記載必須項目**:
   - レビューID（一意の識別子）
   - レビュー対象（対象ファイルの相対パスを明記）
   - 指摘事項（分類、内容、影響/リスク、推奨対策）
   - 総評および次のステップ
5. **指摘事項の修正と完了管理**:
   - レビューで抽出された指摘事項は、必ず「推奨対策」に基づいて対象ファイルへ反映すること。
   - 修正が完了した項目は、レビュー記録表の「ステータス」を `完了` に更新し、「修正内容サマリー」に実施内容を簡潔に記録すること。
   - 指摘事項が複数ファイルに跨る場合、全ての関連ファイルが更新されていることを確認してからステータスを更新すること。

## 目的
設計上の不備や矛盾を早期に発見し、開発のトレーサビリティを確保することで、手戻りの少ない効率的な開発を実現します。

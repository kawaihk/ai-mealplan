# GitHub 作業ルール

## 概要
プロジェクトの透明性を保ち、AIによるコンテキスト把握を容易にするため、GitHub上での操作（イシュー作成、ブランチ管理、プルリクエスト、マージ）に関する標準フローを規定します。

## 1. イシュー (Issue) の作成
- **タイミング**: 新機能の実装、バグ修正、ドキュメント作成など、作業を開始する前に必ず作成する。
- **命名規則**: `[カテゴリ] 概要`
  - 例: `[Feature] ユーザーログイン機能の実装`, `[Bug] 保存ボタンが反応しない問題の修正`
- **記載内容**:
  - **目的/概要**: 何を達成するための作業か。
  - **受け入れ条件 (Acceptance Criteria)**: 作業完了とみなす具体的な基準。
  - **関連資料**: 設計ドキュメント等へのリンク。
- **CLI操作**: `gh issue create --title "[Feature] 概要" --body "内容..."`

## 2. ブランチ管理
- **命名規則**: `カテゴリ/issue#番号-概要`
  - `feature/`: 新機能
  - `bugfix/`: バグ修正
  - `docs/`: ドキュメント作成・更新
  - `refactor/`: リファクタリング
  - 例: `feature/issue#12-user-auth`
- **作成**: 必ず `main` ブランチから最新をプルした状態で作成する。
- **CLI操作**: `gh repo sync` (同期後) -> `git checkout -b "feature/issue#12-auth"`

## 3. コミットメッセージ
- **形式**: `タイプ: 概要 (#イシュー番号)`
  - タイプ: `feat`, `fix`, `docs`, `refactor`, `style`, `test`, `chore`
  - 例: `feat: ログインAPIの実装 (#12)`

## 4. プルリクエスト (PR) の作成
- **タイミング**: 作業が完了し、`.ai/instructions/docs_rule.md` に基づくセルフレビューが完了した時点。
- **記載内容**:
  - **修正内容**: 具体的に何を変更したか。
  - **関連Issue**: `Closes #12` のように記載（マージ時に自動クローズさせる）。
  - **セルフチェックリスト**:
    - [ ] 期待される動作を確認したか
    - [ ] `docs/reviews/` にレビュー記録を作成したか
    - [ ] `README.md` の進捗状況を更新したか
- **CLI操作**: `gh pr create --title "[Feature] 概要 (#12)" --body "修正内容..."`

## 5. レビューとマージ
- **レビュー**: 
  - 他のメンバーまたはAIによるチェックを受ける。
  - 指摘事項はすべて解決（Resolve）した状態にする。
- **マージ方法**: 原則として **Squash and Merge** を使用し、コミット履歴を綺麗に保つ。
- **マージ後の処理**:
  - リモートおよびローカルの作業ブランチを削除する。
  - CLI操作: `gh pr merge --squash --delete-branch`

## 6. AI-Native 特有のルール
- **README同期**: マージ前またはマージ直後に、必ず `README.md` の「進捗状況」を最新化すること（`readme_rule.md` 参照）。
- **ドキュメント整合性**: 実装コードと `docs/` 配下の設計ドキュメントに乖離がないか、PR時に再確認すること。

## 目的
すべての変更がIssueと紐付き、PRを通じて履歴が残ることで、AIが「なぜこの変更が行われたのか」という経緯（コンテキスト）を正確に理解できるようにします。
